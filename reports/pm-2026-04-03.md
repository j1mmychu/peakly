### 3. DEV: Extract VENUES to venues.json — 3-4 hours, P1 performance

app.jsx is 1.95 MB. ~1.4 MB is the VENUES array (3,726 entries). Babel Standalone parses every byte at runtime before React renders anything. On a mid-range Android at 4G: 5-10 seconds of blank screen. The splash screen hides it but the bounce is real.

**Fix:** Move the VENUES array to `venues.json`. On app init, `fetch('./venues.json')` and `setState`. Display shimmer cards while loading. app.jsx stays a single file — this is data extraction, not architecture change.

This is the highest-ROI engineering task available. First-paint time drops ~50%. Mobile conversion improves materially.

**Decision: SHIP this week. No new venues added to app.jsx until venues.json migration ships.**

## One Product Risk Nobody Is Talking About

**The venue expansion from 192 → 3,726 shipped without quality gates. The Content agent found it. The community will find it worse.**

Teahupo'o tagged as "All Levels" is embarrassing but fixable. The deeper problem is that ~800 batch-generated entries have coordinate conflicts with existing named venues, meaning users can favorite one "Cloudbreak" and get alert mismatches on the other. This is a trust problem disguised as a data problem. A user who sets an alert for Teahupo'o, gets a push notification for a beginner-friendly beach break score, and tries to book a flight there is a user who uninstalls and writes a negative review.

**The batch expansion bought launch credibility (3,726 venues vs. competitors) but created a ticking credibility bomb with expert users (the exact community Peakly is targeting).** The fix — deduplication pass removing ~800 coordinate-conflicting entries — takes venue count from 3,726 to ~2,900 but eliminates every factual contradiction. 2,900 accurate venues beats 3,726 contradictory ones. Ship the dedup pass this week, after the P1 bug fixes.
