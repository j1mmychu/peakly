## 4. High Issues (P1) -- Fix This Week

### P1-1: Proxy has ZERO rate limiting

The Express proxy at `server/proxy.js` has no rate limiting middleware. Any bad actor can hammer `/api/flights/latest` and exhaust the Travelpayouts API quota or spike DigitalOcean bandwidth.

**Risk:** API quota exhaustion, upstream bans, VPS CPU/memory spike on 1GB droplet.

**Fix:** Add `express-rate-limit` to proxy.js:

```bash
# On VPS:
cd /path/to/server && npm install express-rate-limit
```

Then add to `server/proxy.js` after `app.use(express.json())`:

```js
const rateLimit = require('express-rate-limit');
const limiter = rateLimit({
  windowMs: 60 * 1000, // 1 minute
  max: 30,             // 30 requests per minute per IP
  standardHeaders: true,
  legacyHeaders: false,
  message: { success: false, error: 'Rate limited. Try again in a minute.' },
});
app.use('/api/', limiter);
```

### P1-2: Flight proxy returning 404 on valid request

`GET /api/flights/latest?origin=JFK&destination=BCN` returned HTTP 404 (157 bytes). The health endpoint is fine (200), so the process is running. Possible causes:

1. Travelpayouts token expired or rotated
2. Upstream Travelpayouts API temporarily down
3. Caddy routing issue (less likely since /health works)

**Fix:** SSH into VPS and check:

```bash
# Check proxy logs
journalctl -u peakly-proxy --since "1 hour ago" | tail -50

# Test locally on VPS
curl -s "http://127.0.0.1:3001/api/flights/latest?origin=JFK&destination=BCN"

# Verify token is set
echo $TRAVELPAYOUTS_TOKEN | head -c 5
```

If the token is expired, get a new one from travelpayouts.com dashboard and restart:

```bash
export TRAVELPAYOUTS_TOKEN=new_token_here
pm2 restart peakly-proxy
```

### P1-3: No `helmet` security headers on proxy

The proxy serves no security headers (X-Content-Type-Options, X-Frame-Options, etc.). While Caddy adds some, Express should add its own defense-in-depth layer.

**Fix:**

```bash
# On VPS:
npm install helmet
```

Add to `server/proxy.js`:

```js

```

## 5. Medium Issues (P2) -- Fix This Sprint

### P2-1: Service worker cache name is stale

`sw.js` has `CACHE_NAME = "peakly-20260330"` but index.html cache buster is `v=20260331a`. The SW precaches `/peakly/app.jsx` without the query param, so it caches whatever CDN version is current rather than the version-busted URL. Not breaking now (stale-while-revalidate handles it), but a deploy could serve stale JS to returning users for up to 24 hours.

**Fix:** Update `sw.js` line 2:

```js
const CACHE_NAME = "peakly-20260405";
```

And bump the cache buster in `index.html` line 346:

```html
<script type="text/babel" src="./app.jsx?v=20260405a" data-presets="react"></script>
```

### P2-2: Push notification icon points to manifest.json

In `sw.js` lines 64-65, the push notification `icon` and `badge` both point to `/peakly/manifest.json` -- a JSON file, not an image. Notifications will show a broken or missing icon.

**Fix:** Use inline SVG data URI (matches apple-touch-icon approach in index.html):

```js
icon: "data:image/svg+xml,<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 192 192'><rect width='192' height='192' rx='38' fill='%230284c7'/><text x='96' y='143' font-size='120' font-family='system-ui' font-weight='700' fill='white' text-anchor='middle'>P</text></svg>",
badge: "data:image/svg+xml,<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 72 72'><rect width='72' height='72' rx='14' fill='%230284c7'/><text x='36' y='54' font-size='44' font-family='system-ui' font-weight='700' fill='white' text-anchor='middle'>P</text></svg>",
```

### P2-3: Alerts stored in-memory only (proxy)

`server/proxy.js` line 196: `const _alerts = new Map()` -- all registered alerts vanish on process restart. With 12 days uptime this hasn't been a problem, but any deploy or crash wipes alert registrations silently.

**Fix (short-term):** Persist alerts to a JSON file:

```js
const fs = require('fs');
const ALERTS_FILE = '/var/lib/peakly/alerts.json';

function loadAlerts() {
  try { return new Map(Object.entries(JSON.parse(fs.readFileSync(ALERTS_FILE, 'utf8')))); }
  catch { return new Map(); }
}
function saveAlerts(map) {
  fs.writeFileSync(ALERTS_FILE, JSON.stringify(Object.fromEntries(map)));
}

const _alerts = loadAlerts();
// Call saveAlerts(_alerts) after every set/delete operation
```

## 8. What Breaks First at Scale

The 1.9 MB app.jsx file transpiled by Babel Standalone in the browser. At current traffic this is invisible -- Reddit users on desktop WiFi won't notice. But as mobile traffic grows past 5K MAU, bounce rates will spike because mid-range Android phones take 5-8 seconds to parse and transpile 1.9 MB of JSX. The total cold-load JS payload is ~5.5 MB (Babel 3.2 MB + app.jsx 1.9 MB + React 150 KB). The fix is to pre-transpile app.jsx at deploy time via a one-line GitHub Action step (`npx babel app.jsx --presets=react --out-file app.js`), then load `app.js` as a regular script -- cutting 3.2 MB of Babel Standalone from the client entirely and eliminating transpilation latency. This takes 15 minutes to implement and should be done before hitting 5K MAU.
