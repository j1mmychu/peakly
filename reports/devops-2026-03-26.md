## Performance Analysis

### File Sizes

| File | Lines | Size | Notes |

| manifest.json | — | 799 B | PWA manifest |

### Performance Concerns

1. **app.jsx is 1.3 MB raw.** Babel Standalone must parse and transpile this in-browser on every cold load. With 2,226 hardcoded venues (~2,000 lines of JSON-like data), initial parse time on mobile could be 3-5 seconds. This is the single biggest performance bottleneck.

2. **Batched weather fetching is well-implemented.** 50 venues per batch with 2s delays. Weather cache (30-min TTL) dramatically reduces repeat API calls. Good.

3. **Flight price deduplication is smart.** Fetches by unique airport code (~100-150 unique airports) rather than per-venue (2,226). Semaphore caps concurrent requests at 3. Well-engineered.

4. **Service worker uses stale-while-revalidate** for app assets, network-first for API calls and CDN. Cache name `peakly-20260326` is current. Good.

5. **No lazy loading for images.** All venue photos are Unsplash URLs with `w=800&h=600` params. Browser handles loading, but 2,226 venue cards in the DOM could cause significant image fetch overhead. Only visible cards should trigger image loads.

### Recommendation

The 1.3 MB JSX file is approaching a pain point for mobile users on slow connections. The venue data (~800 KB of that) could be extracted to a separate JSON file loaded async, but that conflicts with the "no build step, single file" architecture. For now, the service worker cache mitigates repeat-visit load times. Monitor Sentry for performance complaints.
