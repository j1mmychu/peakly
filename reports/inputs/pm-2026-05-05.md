## One Product Risk Nobody Is Talking About

**The share-a-list viral loop is dead code until Jack uses it.**

B.8 is shipped (in code). The K-factor of any viral feature is zero until someone actually shares. Right now the only person on Earth with credentials to send a share is Jack. If Jack doesn't pick 5 friends and send 5 lists this week, the feature has the same impact as not shipping it — except worse, because every future PM/UX agent report will treat it as a live revenue lever and reason about it as if it has signal.

The hidden mistake here is reasoning by spec instead of behavior. The spec says "viral loop." The behavior says "no shares yet." The product call this week is to put one of those two in the driver's seat.

If Jack sends 5 shares and Supabase shows view_count=0 across all of them, the feature is misnamed — it's a "share-with-someone-who-already-installed" link, not a viral acquisition channel, and the launch story shouldn't lean on it. If Jack sends 5 shares and 3 recipients click, the magic-link flow needs to be smoothed before launch (the iOS Safari → magic-link tab-switch pendingImport flow is a fragile chain). Either answer changes the launch order. Neither answer comes from staring at the code.

This is solvable in a week with five iMessage taps. Ship it before the proxy is live in prod and the answer compounds against the timeline.
