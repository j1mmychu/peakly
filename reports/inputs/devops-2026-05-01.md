### P1 — No SRI on CDN Scripts

Three scripts have no `integrity` attribute. A supply-chain attack via a compromised unpkg push injects arbitrary JS into every Peakly session before Sentry or the error boundary can catch it.

```html
<!-- index.html lines 80–82 — missing integrity= on all three -->
<script crossorigin src="https://unpkg.com/react@18.3.1/umd/react.production.min.js"></script>
<script crossorigin src="https://unpkg.com/react-dom@18.3.1/umd/react-dom.production.min.js"></script>
<script src="https://unpkg.com/@babel/standalone@7.24.7/babel.min.js"></script>
```

Generate hashes and add `integrity=` attributes:

```bash
REACT_HASH=$(curl -s https://unpkg.com/react@18.3.1/umd/react.production.min.js | openssl dgst -sha384 -binary | openssl base64 -A)
DOM_HASH=$(curl -s https://unpkg.com/react-dom@18.3.1/umd/react-dom.production.min.js | openssl dgst -sha384 -binary | openssl base64 -A)
BABEL_HASH=$(curl -s https://unpkg.com/@babel/standalone@7.24.7/babel.min.js | openssl dgst -sha384 -binary | openssl base64 -A)
echo "React: sha384-$REACT_HASH"
echo "ReactDOM: sha384-$DOM_HASH"
echo "Babel: sha384-$BABEL_HASH"
```

Then update `index.html`:

```html
<script src="https://unpkg.com/react@18.3.1/umd/react.production.min.js"
  integrity="sha384-REACT_HASH" crossorigin="anonymous"></script>
<script src="https://unpkg.com/react-dom@18.3.1/umd/react-dom.production.min.js"
  integrity="sha384-DOM_HASH" crossorigin="anonymous"></script>
<script src="https://unpkg.com/@babel/standalone@7.24.7/babel.min.js"
  integrity="sha384-BABEL_HASH" crossorigin="anonymous"></script>
```

**Caveat:** SRI blocks the script if the CDN serves different bytes than the hash — including if unpkg serves a cached variant or normalized response. Test in browser after applying. **Time to fix: 20 minutes.**

### P2 — No Content Security Policy

No CSP on `index.html`. GitHub Pages can't set HTTP headers, so a meta tag is the only lever. Babel Standalone requires `'unsafe-eval'`, which limits effectiveness — but blocking unknown script origins still raises the bar.

Add to `<head>` in `index.html`:

```html
<meta http-equiv="Content-Security-Policy" content="
  default-src 'self';
  script-src 'self' 'unsafe-eval' https://unpkg.com https://js.sentry-cdn.com https://plausible.io;
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
  font-src https://fonts.gstatic.com;
  img-src 'self' https://images.unsplash.com data: blob:;
  connect-src 'self' https://peakly-api.duckdns.org https://api.open-meteo.com https://marine-api.open-meteo.com https://plausible.io https://o4511108649058304.ingest.us.sentry.io;
  frame-ancestors 'none';
">
```

**Time to fix: 20 minutes. Test thoroughly — Babel inline eval must still work.**

## 6. P1 — Alert Registrations Lost on Restart

`server/proxy.js` stores alerts in a JavaScript `Map` in process memory. A droplet reboot, pm2 restart, or OOM kill wipes every registered alert. No persistence, no notification to users.

**Fix — add file persistence to proxy.js:**

```js
// Add after the existing _alerts declaration (around line 195):
const ALERTS_PERSIST_PATH = process.env.ALERTS_PATH
  || path.join(__dirname, 'data', 'alerts.json');

function _loadAlerts() {
  try {
    const raw = fs.readFileSync(ALERTS_PERSIST_PATH, 'utf8');
    const entries = JSON.parse(raw);
    for (const [k, v] of entries) _alerts.set(k, v);
    console.log(`[alerts] loaded ${_alerts.size} alerts from disk`);
  } catch {}
}

function _saveAlerts() {
  try {
    fs.mkdirSync(path.dirname(ALERTS_PERSIST_PATH), { recursive: true });
    fs.writeFileSync(ALERTS_PERSIST_PATH, JSON.stringify([..._alerts.entries()]));
  } catch (e) { console.error('[alerts] save failed:', e.message); }
}

_loadAlerts();
```

Then after `_alerts.set(alertId, record)` in `POST /api/alerts`:
```js
_alerts.set(alertId, record);
_saveAlerts();
```

And after `_alerts.delete(alertId)` in `DELETE /api/alerts/:alertId`:
```js
_alerts.delete(alertId);
_saveAlerts();
```

This is a VPS-side change — requires SSH to deploy. **Time to fix: 30 minutes including deploy.**

Separate from persistence: there is still **no polling worker**. Alerts are registered and stored but never evaluated. Users who set a strike alert get silence. This is a P1 feature gap, not a DevOps gap — tracked in CLAUDE.md.

A single post on Reddit/HN with 300 simultaneous cold-cache visitors could blow the account before any real MAU growth. This is the highest-impact infrastructure debt by a significant margin.

**Mitigation (server-side weather cache in proxy.js):**

```js
// Add to server/proxy.js — collapses N client requests to 229 cached proxy requests
const wx_cache = new Map();
const WX_TTL = 2 * 60 * 60 * 1000; // 2hr

app.get('/api/weather', async (req, res) => {
  const { lat, lon, type, params } = req.query;
  if (!lat || !lon) return res.status(400).json({ error: 'lat and lon required' });
  const flat = parseFloat(lat), flon = parseFloat(lon);
  if (isNaN(flat) || isNaN(flon)) return res.status(400).json({ error: 'invalid lat/lon' });
  const key = `${flat.toFixed(3)},${flon.toFixed(3)},${type || 'wx'}`;
  const cached = wx_cache.get(key);
  if (cached && Date.now() - cached.ts < WX_TTL) {
    res.setHeader('X-Cache', 'HIT');
    return res.json(cached.data);
  }
  const base = type === 'marine'
    ? 'https://marine-api.open-meteo.com/v1/marine'
    : 'https://api.open-meteo.com/v1/forecast';
  const url = `${base}?latitude=${flat}&longitude=${flon}${params ? '&' + decodeURIComponent(params) : ''}`;
  try {
    const { status, json } = await fetchJson(url);
    if (status !== 200) return res.status(status).json(json);
    wx_cache.set(key, { data: json, ts: Date.now() });
    res.setHeader('X-Cache', 'MISS');
    return res.json(json);
  } catch (err) {
    return res.status(502).json({ error: err.message });
  }
});

setInterval(() => {
  const cutoff = Date.now() - WX_TTL;
  for (const [k, v] of wx_cache) if (v.ts < cutoff) wx_cache.delete(k);
}, WX_TTL);
```

**This is the single highest-ROI infrastructure change.** It costs $0, runs on existing VPS, and prevents an Open-Meteo ban that would silently zero out all venue scores for every user. Estimated implementation: 4 hours including client-side changes to point `fetchWeather`/`fetchMarine` at the proxy.

## 8. What Breaks First at Scale

The client-side weather fetch is the structural failure at 500 MAU. 229 venues × 2 API types = 458 calls per cold-cache user. GitHub Pages serves all users from ~50 shared edge IPs globally. Open-Meteo will rate-limit those IPs. When it does, `fetchWeather` returns null, `scoreVenue` can't compute, every card shows 0 or a stale score, and no error is thrown anywhere visible to the user. The app just silently stops working. No alerting fires. No Sentry event. Users see a broken explore feed and leave.

**The fix is a server-side weather cache in proxy.js.** It's 50 lines of Node. At that point, all 5K users share the same 229 cached weather responses and Open-Meteo never sees more than 229 requests per 2hr window per deployment. Build this before the first traffic spike.

Second failure: the 1GB VPS will OOM under sustained load. The in-memory rate limiter Map, alert Map, and Node.js heap will approach the 1GB ceiling. `pm2 monit` should be running and alerting. Upgrade to the $12/mo 2GB plan before any launch push.
