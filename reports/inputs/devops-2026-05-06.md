## P1 — NEW: Supabase JS is loaded eagerly, nullifying the documented lazy-load

**Severity:** P1 — perf regression on every cold load, breaks a documented contract.

`app.jsx:18-22` comment block:
> "The Supabase JS lib itself loads LAZILY (not at first paint) — only when a returning user has a saved session, a magic-link redirect needs handling, or the user taps "Sign in" in Profile. Saves anonymous-visitor first-paint payload (~80KB)."

`app.jsx:48 ensureSupabase()` correctly injects a `<script>` tag on demand. But `index.html:85` *also* loads the same UMD bundle eagerly:

```html
<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.45.4/dist/umd/supabase.min.js"></script>
```

Net effect: every anonymous visitor downloads ~80 KB gzipped of Supabase JS at first paint, including users who never sign in. The promised saving is zero. CLAUDE.md still claims the library is lazy-loaded — that doc is now wrong on a shipped feature.

**The fix is a 3-line delete in `index.html`** (lines 83–85):

```diff
--- a/index.html
+++ b/index.html
@@ -80,10 +80,6 @@
   <script crossorigin src="https://unpkg.com/react@18.3.1/umd/react.production.min.js"></script>
   <script crossorigin src="https://unpkg.com/react-dom@18.3.1/umd/react-dom.production.min.js"></script>

-  <!-- Supabase JS (UMD) — cloud sync for wishlists/alerts/profile.
-       Inert until SUPABASE_URL is set in app.jsx. -->
-  <script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.45.4/dist/umd/supabase.min.js"></script>
-
   <!-- Babel standalone — transforms JSX in the browser -->
   <script src="https://unpkg.com/@babel/standalone@7.24.7/babel.min.js"></script>
```

`ensureSupabase()` already handles the cold path: when `typeof supabase === "undefined"` it injects the script tag itself. After deletion, the bundle only loads for: returning users with a saved session, magic-link callbacks, and users tapping Sign in — exactly the contract the comment promises.

**Risk:** Low. Touches one file, deletes 3 lines, no app.jsx change required. Will not affect first-paint react/babel rendering. Bumps cache-buster in lockstep is wise but not strictly required (HTML is fetched network-first per `sw.js:25-32`).

**Estimated apply time:** 30 seconds. After applying, bump `PEAKLY_BUILD` and `CACHE_NAME` to `20260506a` to force one round of fresh `index.html` for users whose service worker is mid-fetch.

Why I'm not putting this in `reports/ready-to-ship/`: the delete-tag change-class isn't in the Rule 1 enumeration ({flip-boolean, add-condition, update-string, swap-color, change-fontSize}). Filing it as a finding with inline diff so Jack reads it once and decides.

## P1 — NEW: Unsplash venue photos miss `auto=format`/`q=` — every photo ships as JPEG

All 155 venue photos in `app.jsx` request from Unsplash with `?w=800&h=600&fit=crop` and (for some) `&fp-x=…&fp-y=…`. None of them include `&auto=format` or `&fm=webp`, and none cap quality. Unsplash defaults to JPEG @ q≈80 in that case.

Adding `&auto=format&q=75` (Unsplash's documented "send WebP/AVIF when supported, fall back to JPEG, target 75% quality") would cut typical card-photo bytes by 40–55% on mobile. At 155 photos × ~50 KB savings each, that's roughly 7 MB of bandwidth saved per cold-load if the user scrolls the full list — concretely, the difference between Explore feeling instant and Explore feeling sluggish on coach-class wifi.

**One-shot sed** (run from repo root, then verify the diff before commit):

```bash
cd ~/peakly
sed -i.bak -E 's|(images\.unsplash\.com/photo-[^"]*?w=800&h=600&fit=crop)([^"]*)"|\1\2\&auto=format\&q=75"|g' app.jsx
# inspect:
git diff app.jsx | head -40
# if good:
rm app.jsx.bak
# bump cache key (sw.js + app.jsx PEAKLY_BUILD + index.html ?v=) to 20260506a
```

The pattern is conservative: it only touches Unsplash URLs that already contain `w=800&h=600&fit=crop` (which is all 155 of them per `grep -oE "w=[0-9]+&h=[0-9]+" app.jsx | sort | uniq -c` → 155 hits, all `w=800&h=600`).

**Risk:** Low — Unsplash respects `auto=format` server-side, no client behavior change. First load after deploy will re-fetch all photos because URLs change; for already-warm caches this is a one-time cost.

**Estimated apply + verify time:** 4 minutes including a one-page eyeball of Explore on iPhone Safari to confirm photos still render.

Why not Rule 1: 155 lines edited via sed is not "one-line, scoped." Filing as a finding.
