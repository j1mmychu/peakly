**Design Score: 6.5/10** (up from 6/10 — deal-based flight scoring and consistent sort order improve trust; onboarding wall and touch target issues remain)

## Top 3 Visual Issues

### 1. Emoji in Category Pills Violates Design Decision

**Severity: Medium | File: app.jsx | Line: ~6879**

CLAUDE.md explicitly states: "Remove emoji from UI chrome. Emoji in category pills, section headers, and badges look amateur. Clean text + SVG only." Yet category pills still render emoji via `{c.emoji} {c.label}`. The onboarding screen also uses emoji icons (phone icon, lightning bolt, bell) rendered as styled divs, which is fine — but the Explore tab pills break the rule.

**Current code (line 6879):**
```jsx
{c.emoji} {c.label}
```

**Fix:**
```jsx
{c.label}
```

This also recovers horizontal space in the pill bar, which is tight at 430px with 4 pills + a saved count pill.

### 2. Duplicate Flight Price Display on ListingCard

**Severity: Medium | File: app.jsx | Lines: ~5780 and ~5845**

ListingCard shows the flight price TWICE: once as a badge overlay on the photo ("from $X") and again in the card body with strikethrough pricing. This is visually cluttered and creates cognitive overhead — especially because the photo badge and body price can show different formatting (badge is 10px, body is 14px). It dilutes the impact of both.

**Fix:** Remove the flight price badge from the photo overlay on ListingCard. Keep only the body price with the strikethrough. The GoVerdictBadge (GO/MAYBE/WAIT) on the photo is enough. The FeaturedCard can keep the photo badge since it uses "% off" language which is additive, not duplicative.

```jsx
// Lines 5771-5789: Remove the flight price + trending badges from photo overlay
// Keep only GoVerdictBadge:
<div style={{ position:"absolute", top:12, left:12, display:"flex", gap:5, alignItems:"center" }}>
  {listing.conditionLabel !== "Checking conditions…" && <GoVerdictBadge score={listing.conditionScore} />}
</div>
```

### 3. "Book" Button Uses Blue Gradient — Conflicts with CTA Color Decision

**Severity: Medium | File: app.jsx | Lines: ~5854, ~5935**

CLAUDE.md records: "CTA buttons use #222 (dark) not blue. Feels more premium." But the "Book" button on both ListingCard and FeaturedCard uses `background:"linear-gradient(135deg,#1a56db,#0ea5e9)"` — a blue gradient. This directly contradicts the design decision. The search button in the header is also blue (#0284c7), creating visual competition.

**Fix (ListingCard line ~5854):**
```jsx
background:"#222", borderRadius:20,
```
**Fix (FeaturedCard line ~5935):**
```jsx
background:"#222", borderRadius:20,
```

The dark CTA will also improve hierarchy — blue should be reserved for the brand wordmark and interactive links, not buttons.

## Top 3 Interaction Issues

### 1. Share Button Touch Target Still 32x32px (Flagged Yesterday, Not Fixed)

**File: app.jsx | Line: ~5745**

The share button in ListingCard is 32x32px. Apple HIG and WCAG 2.5.5 require 44x44px minimum. This was flagged in the April 5 report. Users on mobile will mis-tap the underlying card instead of hitting the share button.

**Fix:**
```jsx
width:44, height:44,
```

Also increase the heart button on CompactCard (line ~5971) from 36x36 to 44x44, and add `padding:4px` around the icon so the visual size doesn't change but the hit area expands.

### 2. Category Pills Total Height ~28px — Below Touch Target Minimum

**File: app.jsx | Line: ~6872**

Category pills have `padding:"5px 6px"` making total rendered height ~28px. At the 4px gap between pills, the effective touch area is ~32px. This is 27% below the 44px minimum. On small phones (iPhone SE, older Android), this becomes a usability issue.

**Fix:**
```jsx
padding:"8px 10px",
```
This gets rendered height to ~36px. Combined with the gap, effective target is ~40px — acceptable for grouped pill controls per Material Design specs.

### 3. BottomNav Tab Labels at 10px — Below WCAG Minimum

**File: app.jsx | Line: ~10405**

BottomNav tab labels are `fontSize:10`. At this size, the text "Explore", "Alerts", "Profile" is difficult to read, especially for users with any visual impairment. iOS tab bars use 10pt but that's 13.3px at 1x — and Apple's own HIG recommends against going below 11pt (14.7px) for critical navigation.

**Fix:**
```jsx
fontSize:11,
```

## Accessibility Concerns

1. **Low contrast timestamp** — "Updated Xm ago" text at `color:#bbb` on white (#fff). Contrast ratio: ~1.9:1. Fails WCAG AA. Change to `#888` (4.5:1). (Line ~7089)

2. **GoVerdictBadge "MAYBE" text** — Yellow (#eab308) on light yellow (#fef9c3). Contrast ratio: ~2.1:1. Fails WCAG AA for small text. Change MAYBE text color to `#a16207` (darker amber) for 4.5:1+ ratio. (Line 5661)

3. **Missing aria-labels on search bar** — The SearchBar pill shows "Anywhere · Any time · New York" but has no aria-label describing its function. Screen readers won't know it opens a search sheet. Add `aria-label="Open search filters"`.

4. **Flight price "est." label** — `color:#888` at `fontSize:10`. Barely readable. This label carries trust information (estimated vs. live price). Bump to `fontSize:11` and `color:#666`.

## User Flow Analysis

**Primary flow: Land → Browse → Tap venue → View details → Book flight**

| Step | Taps | Notes |
|------|------|-------|
| Splash screen | 0 (auto) | 1.8s minimum visible — good |
| Onboarding step 0 | 1 | "Get Started" — still a wall before any content |
| Airport picker | 2-3 | Select airport + "Find my flights" |
| Explore loads | 0 | Weather fetches, shimmer skeletons shown |
| Tap venue card | 3-4 | First venue with conditions visible |
| VenueDetailSheet opens | +1 | Photo hero + conditions + flight price |
| Tap "Book" → Aviasales | +1 | Deep link opens in new tab |

**Total: 5-6 taps to first venue detail. 6-7 taps to booking. Down from yesterday's 6-7/7-8 thanks to the airport picker modal consolidating steps.**

**Remaining friction:** The onboarding wall (steps 0 and 1) still blocks content. The ideal flow puts content first — show the Explore tab immediately with JFK as default, and prompt for airport change via an inline banner. This was noted yesterday. Repeating because it's the single highest-impact UX change remaining.

## Inspiration

**Steal from Surfline's session planner:** Surfline shows a weekly bar chart of conditions (swell + wind + tide) for a single spot — at a glance, you see which day is best. Peakly has the "Best Window" badge on cards, but the VenueDetailSheet's 7-day forecast is a text table. Converting it to a mini bar chart (colored by GO/MAYBE/WAIT thresholds) would make the detail sheet feel significantly more premium and data-rich, matching the quality bar of Surfline's daily breakdown.

---

## Decision Made

**Remove duplicate flight price from ListingCard photo overlay.** The price shown as a badge ON the photo AND in the card body is redundant and clutters the card. Keep only the body price with strikethrough. The photo overlay should carry only the GO/MAYBE/WAIT verdict and the condition label — these are the high-signal, glanceable elements. Flight price is a conversion detail that belongs in the body alongside the Book CTA. This simplifies the card visual hierarchy: photo = conditions at a glance, body = details + action.
