| Plus Jakarta Sans (font) | ~60 KB | Subset via Google Fonts |
| **Total page payload** | **~3.25 MB** | **P1** |

### Single Largest Performance Bottleneck: Babel In-Browser Transpilation

**This is the bottleneck.** Every page load forces the browser to:
1. Download 1MB Babel Standalone
2. Parse the 2MB app.jsx file
3. Transpile ~11,000 lines of JSX to JS in the main thread

On a mid-range Android phone (Snapdragon 665, 4GB RAM), this is **2–4 seconds of blocking JS** before any React renders. On a budget phone, 4–8 seconds. This directly tanks Lighthouse score and first-contentful-paint.

**Fix options (in order of effort):**
- **Option A — Pre-transpile app.jsx at deploy time (2 hours effort):** Add a GitHub Actions step that runs `npx babel app.jsx -o app.js` and serves the pre-transpiled file. Drop Babel from the CDN entirely.

  ```yaml
  # .github/workflows/deploy.yml — add this build step before Pages deploy
  - name: Transpile JSX
    run: |
      npm install -g @babel/cli @babel/preset-react @babel/core
      babel app.jsx --presets @babel/preset-react -o app.js
  ```

  Then in `index.html`, change:
  ```html
  <!-- REMOVE -->
  <script src="https://unpkg.com/@babel/standalone@7.24.7/babel.min.js"></script>
  <!-- CHANGE type and src -->
  <script src="./app.js?v=20260410a"></script>
  ```

  **This eliminates the 1MB Babel download AND the 2–4s transpile time. Biggest win available.**

- **Option B — Do nothing until 1K users validate the product (recommended for now).**

### Missing: Subresource Integrity (SRI) on CDN Scripts

All three CDN scripts (React, ReactDOM, Babel) load without `integrity` hash attributes. If unpkg CDN is compromised, malicious scripts load with full trust. Low risk for a startup, but a hardening opportunity.

**Fix (add to `index.html`):**
```html
<!-- Generate hashes: openssl dgst -sha384 -binary react.production.min.js | openssl base64 -A -->
<!-- Then add integrity="sha384-<hash>" crossorigin="anonymous" to each script tag -->
```
Effort: 30 minutes. Priority: P2 (post-launch).

## What Breaks First at Scale

**Babel in-browser transpilation collapses first.** Not because of a server limit — GitHub Pages and the DigitalOcean proxy will handle 10K concurrent users fine. What breaks is user-perceived performance: as app.jsx grows past 15K lines, cold-start parse time on mid-range Android phones hits 5–8 seconds. Bounce rates spike, Reddit launch underperforms. Prevention: pre-transpile app.jsx in the GitHub Actions deploy pipeline before the Reddit launch. Eliminates the 1MB Babel download and the 2–4s main-thread block simultaneously. Two-hour fix with surgical changes to `deploy.yml` and `index.html`. Everything else — HTTPS proxy, weather caching, flight deduplication — is sound and will scale to 10K MAU without modification.
