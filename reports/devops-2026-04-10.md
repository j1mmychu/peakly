## Security Review

**Status: CLEAN in HEAD. One historical issue. One unmitigated attack surface.**

| Check | Result |
|---|---|
| API tokens in HEAD tree | CLEAN |
| Travelpayouts token | Server-side only (`server/proxy.js` reads from `process.env.TRAVELPAYOUTS_TOKEN`) |
| Push tokens | localStorage only (`peakly_push_token`) — acceptable |
| Recent commits (last 10) | No secrets |

**Historical leak (low severity, not actionable without history rewrite):** A **Duffel test token** (`duffel_test_...`) lives in git history from the initial commit through `f25c801` ("Wire Travelpayouts real flight prices + remove Duffel"). It's a `_test_` credential (non-production), so the blast radius is small, but anyone running `git log -p` could find it. Options: rotate the Duffel test key at their dashboard (5 min), or accept the risk. I recommend rotating.

**Unmitigated attack surface — VPS proxy has no app-level rate limiting:** `server/proxy.js` handles 429s *from upstream* (Travelpayouts) but has **no limiter on inbound requests**. A single bad actor could hammer `/api/flights/latest` and burn through the Travelpayouts monthly quota, breaking flight pricing for real users. Fix is ~10 lines:

```js
// Add to server/proxy.js after express init
const rateLimit = require('express-rate-limit');
app.use('/api/flights/', rateLimit({ windowMs: 60_000, max: 60 })); // 60 req/min/IP
```

Requires `npm i express-rate-limit` in `server/` and a proxy restart. **This is the #1 infra risk for ProductHunt week** — if the launch trends and someone scripts it for fun, the flight feature goes dark.
