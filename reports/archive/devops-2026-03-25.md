| Metric | Value | Assessment |
|--------|-------|------------|
| app.jsx file size | 412 KB (6,134 lines) | Large but acceptable for single-file architecture |

| Babel transpilation | Runtime, every page load | Known trade-off; ~200-400ms on modern devices |
| Unsplash images | ~192 venue photos (external CDN) | Loaded on-demand via `w=800&h=600` params, good |

**Key findings:**

**Weather fetch is unthrottled.** On page load, the app fires `Promise.allSettled()` across all 192 venues simultaneously (line 5843-5856). That's 192 weather requests + ~80 marine requests = **~272 concurrent API calls to Open-Meteo on every page load.** At 30 concurrent users, that's 8,160 requests per page load cycle. Open-Meteo free tier is 10,000/day. **This will blow past rate limits with even modest traffic.**

**No weather caching exists.** Grep for localStorage weather cache, TTL, or any caching mechanism returned zero results. Weather data refreshes every 10 minutes (line 5872) — another 272 calls each cycle.

**Flight API is well-optimized.** Deduplication by airport code reduces ~192 venue lookups to ~15 unique airport calls, batched in groups of 5 with 300ms delays (lines 5882-5908). Good engineering.

**Service worker concern:** `CACHE_NAME` is still `"peakly-v1"` (sw.js line 2). CLAUDE.md mentions a cache buster bump to `v=20260325c` on the HTML script tag, but the SW cache name hasn't been versioned. If the SW serves stale app.jsx from cache, users could get old code. The stale-while-revalidate strategy (line 30-40) helps, but a cache name bump would force a clean slate.

## 5. Service Worker Risk Assessment

The SW uses stale-while-revalidate, which means:
1. First visit: fetches from network, caches it
2. Return visit: serves cached version immediately, fetches update in background
3. Third visit: gets the updated version

This is generally fine, but caused a known outage previously (per CLAUDE.md). The current `CACHE_NAME = "peakly-v1"` has never been bumped. If a bad deploy gets cached, users are stuck until they manually clear cache or the SW updates in the background.

**Recommendation:** Bump `CACHE_NAME` to `"peakly-v2"` on the next deploy to force cache invalidation. Consider adding a version check mechanism.
