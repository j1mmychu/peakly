**Medium-risk hardening still open** (flagged in CLAUDE.md P1 list, not touched this run):

1. **No SRI hashes** on Sentry / React / ReactDOM / Babel `<script crossorigin>` tags. If unpkg/Sentry CDN is ever compromised, we ship their payload. Fix: add `integrity="sha384-..."` to each. Risk of fix: zero if the hash is right, breakage if wrong. Worth doing.
2. **No `Content-Security-Policy` meta tag.** Adding a proper CSP is tricky because Babel Standalone uses `eval`/`Function()` which blows up strict CSP. Workaround is `'unsafe-eval' 'unsafe-inline'` for script-src, which is a watered-down CSP but still better than nothing (blocks most XSS vectors from other origins). **Recommendation: ship a permissive CSP now, tighten post-launch when we cut Babel.**

**Not a security issue but worth noting:** `/api/alerts` is currently a write endpoint with in-memory storage and no auth — anyone who knows the URL can register an alert with an arbitrary `alertId` and push token. Low abuse potential today (no worker actually reads it yet) but becomes a vector the moment the worker is wired. Add a simple HMAC or origin check when you build the worker.

## 5. Decision Made

**Add SRI hashes to all three `<script>` CDN tags in `index.html` this week.** Single-line change per tag, zero runtime cost, closes the highest-likelihood supply-chain vector we currently have exposed. No action taken this pass — flagging for Jack to greenlight because it's touching `index.html` load order.

To compute the hashes Jack can just run:

```
for url in \
  https://unpkg.com/react@18.3.1/umd/react.production.min.js \
  https://unpkg.com/react-dom@18.3.1/umd/react-dom.production.min.js \
  https://unpkg.com/@babel/standalone@7.24.7/babel.min.js \
  https://js.sentry-cdn.com/9416b032a46681d74645b056fcb08eb7.min.js ; do \
  echo "$url → sha384-$(curl -s $url | openssl dgst -sha384 -binary | openssl base64 -A)"; \
done
```

Then paste each hash into the matching `<script>` tag as `integrity="sha384-..."`.
