**Bottlenecks identified:**
- **Runtime Babel transpilation** is the biggest first-load cost. Babel Standalone (~600 KB) + parsing 474 KB of JSX adds ~1-3s cold-start on slow connections. Not fixable without a build step (deliberately avoided per architecture).
- **Weather batching**: 231 venues fetched in batches of 50 with 2s delays = up to ~9 seconds to fully populate on first load. This is intentional and already handled with skeleton loaders.
- **Images**: All `<img>` tags use `loading="lazy"` ✅ — good call.
- **No virtualization** on the venue list. At 231 venues rendering all cards simultaneously, this could cause jank on low-end devices. Not critical at current count; watch at 500+ venues.

### ⚠️ Issue 1 (Known): No SRI on CDN scripts
`react`, `react-dom`, and `babel` are loaded from `unpkg.com` without `integrity=` hashes. A compromised CDN could inject malicious code. Medium risk — apply SRI post-launch. Adding SRI to Babel Standalone could break inline JSX evaluation; needs careful testing.

### ⚠️ Issue 2 (Known): No CSP meta tag
No `Content-Security-Policy` header or meta tag. Low-medium risk on a GitHub Pages static site. Post-launch hardening item.

### 🐛 NEW BUG: Duplicate `POST /api/waitlist` route in `proxy.js`

**Lines 234 and 343 both register `app.post('/api/waitlist', ...)`.**

Express uses the first matching handler — the second one (which is the better implementation) is dead code and never executes.

**First handler (line 234) — ACTIVE (but inferior):**
- Validates with `email.includes('@')` (weak)
- Records: `{ email, joinedAt }`
- No source tracking, no IP logging

**Second handler (line 343) — DEAD CODE (but superior):**
- Validates with proper regex `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`
- Records: `{ email, source, ip, at }` — includes IP and source for attribution
- Uses `WAITLIST_PATH` env override (more flexible)
- Creates directory recursively at startup (safer)

**Fix:** Remove lines 228–251 (first handler + `WAITLIST_FILE` const). Keep the second handler. No functional change to production behavior for current signups — first handler was working fine — but the superior second handler's source/IP tracking will activate.
