## Top 3 Visual Issues

### 1. Card-tile Book button is blue — VenueDetailSheet sticky CTA is dark (#222) ✓

There's a contradiction. The sticky flight CTA at the bottom of `VenueDetailSheet` (L5909) correctly uses `background:"#222"`. The Book buttons on `ListingCard` (L2124) and `FeaturedCard` (L2209) still use `linear-gradient(135deg,#1a56db,#0ea5e9)`.

The inconsistency is arguably worse than either direction alone. Users see a blue "Book" on cards and a dark "Flights" in the detail sheet — two different CTAs for the same action, styled differently.

**Fix — 2 lines, under 2 minutes:**
```jsx
// L2124 (ListingCard Book button):
background:"#222", borderRadius:20, padding:"8px 14px", minHeight:44,

// L2209 (FeaturedCard Book button):
background:"#222", borderRadius:20, padding:"8px 14px", minHeight:44,
```
While you're there: bump `minHeight` from 36 → 44 (Apple HIG minimum for tap targets). Same two lines.

### 2. Score color system has three competing implementations

`getGoVerdict()` (L1926): green `#22c55e` / yellow `#eab308` / red `#ef4444` — used in card badges.

`ScoreDot` (L1958): five-tier spectrum including `#84cc16` and `#f97316` — used in list rows.

`VenueDetailSheet` conditions panel (L5668): `#ff385c` (Airbnb red) / `#ea580c` (orange) / `#555` (grey) — used inside the detail score tile.

Three color systems for one concept. Users build trust by seeing consistent encoding — "green means go" breaks down when green means different things in different components.

**Decision rule:** GoVerdict palette (`#22c55e/#eab308/#ef4444`) is the canonical system. Kill ScoreDot's 5-tier, kill the detail panel's `#ff385c`. One palette, three states.

### 3. `opacity:0` image fade pattern silently breaks on cached photos

All four photo `<img>` elements (L1996, L2151, L2233, L5603) start at `opacity:0` and rely on an `onLoad` event to fade in. The `ref` callback is supposed to handle already-cached images:
```jsx
ref={img => { if (img && img.complete) img.style.opacity = 1; }}
```
The bug: Babel processes JSX top-down. The `style={{opacity:0}}` inline prop is set BEFORE the `ref` fires. On a cached image, `img.complete` may already be true, but the style hasn't been applied yet — so the ref check happens on the DOM element before the style is painted. On some devices/browsers this leaves cached images invisible forever (no re-trigger of `onLoad` for already-cached images in Chrome).

**Fix:**
```jsx
// Replace the ref pattern with:
onLoad={e => { e.target.style.opacity = 1; e.target.style.transform = "scale(1)"; }}
// Add to img style:
style={{ ..., opacity: 0, transition: "opacity 0.35s ease" }}
// And on mount via useEffect or a state bool — or simplest: use CSS `loading="eager"` for hero, `lazy` for grid
```

Safest short fix for all 4 sites:
```jsx
ref={img => { if (img) { if (img.complete && img.naturalWidth > 0) { img.style.opacity = 1; } } }}
```
The `img.naturalWidth > 0` check confirms the image actually loaded (not just that the element exists).

## Top 3 Interaction Issues

### I1. Share button is 32×32 — smallest touch target in the app

L2011: `width:32, height:32` on the share button in `ListingCard`. Apple HIG / Google Material minimum: 44px. This is the smallest tap target in the app. On a thumb-based mobile session at any scroll speed this gets missed or mis-tapped constantly.

```jsx
// L2011:
width:38, height:38,  // at minimum; ideally 44
```
Also: the share button uses `↑` text as its icon. That's the up arrow (U+2191), not a share icon. Every platform uses a different share glyph for this. Users won't recognize it. Use an SVG share icon or the native `navigator.share()` affordance.

### I2. "Best window right now" hero label is lowercase except in specific states

L3290: `textTransform:"uppercase"` + `letterSpacing:"0.06em"` for "Your best window right now" when photo is shown. But at L3305 (no-photo fallback): same label is rendered without `textTransform:"uppercase"`. Two different visual treatments for the same label based on whether a photo exists. This creates visual instability as weather loads.

Consistent rule: uppercase tracking label should be applied regardless of photo state.

### I3. Venue detail sheet: flight CTA depth varies by content

The sticky flight CTA bar is pinned to the bottom of the sheet (L5895+) — that's good. But the in-scroll "Set Alert" button (L5687) appears before the forecast, and the forecast (7 days of data) can be 200px tall. On a short-viewport device, users see the alert CTA and the forecast before they see the hotel/flight sticky bar. The scroll depth to "see" flight price data without the sticky bar is 3–4 viewport heights.

The two-column "Conditions / Flight" block at L5664 is the right approach — it's near the top and shows the price prominently. The sticky bar then handles the actual booking action. That architecture is good. The gap is: the sticky bar has an ✈️ emoji on it (L5912). Remove it. The dark background + "Flights" label + price is already sufficient signal. Emoji dilutes the premium feel.
