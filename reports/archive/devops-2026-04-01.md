### Performance Concerns

1. **1.95 MB JSX file transpiled at runtime by Babel** — this is the single biggest perf risk. On a mid-range phone, Babel parsing ~2MB of JSX takes 3-8 seconds. Users see the splash screen during this time, but it's still slow.

2. **3,726 venues in a JS array** — approximately 1.2-1.4 MB of the file is venue data. This gets parsed into memory on every page load. Consider: could venue data be a separate JSON file loaded async after initial render?

3. **Batched weather fetching (50/batch, 2s delay)** — smart rate-limit protection, but means full weather data takes ~150 seconds to load for all venues. Pagination + priority fetch mitigates UX impact.

4. **No code splitting possible** — single-file architecture means everything loads upfront. This is a known tradeoff per CLAUDE.md.

### Recommendation
The 1.95 MB file is approaching a threshold where mobile users on 3G will experience 5-10 second load times before Babel even starts parsing. **If venue count grows further, consider extracting VENUES data into a separate `venues.json` loaded via fetch() after initial render.** This doesn't violate the single-file React architecture — it's just data.

### Security Concern: No Server-Side Rate Limiting on Proxy

The proxy at `server/proxy.js` forwards 429 errors from Travelpayouts upstream, but has **no rate limiting of its own**. An attacker could:
- Flood `/api/flights` or `/api/flights/latest` to exhaust the Travelpayouts API quota
- Flood `/api/alerts` POST endpoint to fill server memory (alerts stored in-memory)
- DDoS the 1GB RAM VPS

**Recommendation:** Add `express-rate-limit` to `server/proxy.js`:
```js
const rateLimit = require('express-rate-limit');
app.use('/api/', rateLimit({ windowMs: 60000, max: 60 })); // 60 req/min per IP
```
This is a 5-minute fix that prevents quota exhaustion and basic DDoS.
