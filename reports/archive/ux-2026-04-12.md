## Top 3 Visual Issues (code audit, since live site is down)

### 1. Hero/Card Photo Overlay Still Too Dark
- **ListingCard** line 1866: `rgba(0,0,0,0.55)` gradient overlay
- **VenueDetailSheet** line 2103: `rgba(0,0,0,0.58)` gradient overlay
- **Best Right Now** line 3149: `rgba(0,0,0,0.5)` gradient overlay
- Airbnb uses 0–0.35. Our overlays crush dark photos — venue titles become unreadable on night/dusk shots. The photo IS the product; stop hiding it.

**Fix:**
```jsx
// ListingCard (line 1866)
background:"linear-gradient(to top, rgba(0,0,0,0.38) 0%, transparent 52%)"

// VenueDetailSheet (line 2103)
background:"linear-gradient(to top,rgba(0,0,0,0.35) 0%,transparent 50%)"

// Best Right Now (line 3149)
background:"linear-gradient(to top, rgba(0,0,0,0.35) 0%, transparent 60%)"
```

### 2. Touch Targets Below 44px Minimum
Multiple interactive elements fail Apple's Human Interface Guidelines:
- Share button: 32x32px
- Filter clear X buttons: 18-20px
- Date input fields: ~36px effective height
- GoVerdictBadge dots: 6x6px (sm size on grid cards)

These feel broken on mobile. Users miss taps, get frustrated, leave.

### 3. No Visible Focus Indicators on Inputs
All text inputs use `outline: none` with no replacement focus ring. Screen reader and keyboard users get zero feedback. Tab-navigation is blind.

**Fix:** Add to global styles:
```css
input[type=text]:focus-visible, input[type=email]:focus-visible {
  outline: 2px solid #0284c7;
  outline-offset: 1px;
}
```

## Top 3 Interaction Issues

### 1. No Loading Skeleton for Hero Card
"YOUR BEST WINDOW RIGHT NOW" renders with partial data. The hero — the single most important element on the page — shows a WAIT badge as its first impression while weather data loads. Users see the app's #1 recommendation telling them to wait.

**Component:** ExploreTab, hero section (~line 3100)
**Fix:** Show SkeletonCard variant for hero until first venue has weather data loaded.

### 2. Border-Radius Inconsistency Across Components
Counted 8 different border-radius values in active use: 8, 10, 12, 14, 16, 18, 20, 28px. This creates visual noise — nothing feels like it belongs to the same system.

**Decision:** Standardize to 3 values:
- Small: 8px (tags, badges, inputs)
- Medium: 14px (cards, buttons)
- Large: 24px (sheets, modals)

### 3. Condition Loading Text Uses #aaa Color
The loading placeholder text for condition scores uses #aaa on white (#fff) background. Contrast ratio: 2.32:1. WCAG AA requires 4.5:1 for normal text.

**Fix:** Use #888 minimum for any text on white backgrounds.

## Inspiration

**AllTrails "Trail Conditions" badges** — they use a single, bold horizontal bar that fills proportionally (green → yellow → red) instead of a numeric score. No explanation needed. The user glances and knows. Peakly's score numbers (72, 85, 91) mean nothing without context. A visual bar that reads "PERFECT → GOOD → MEH → SKIP" would eliminate the onboarding problem flagged in CLAUDE.md item #1.
