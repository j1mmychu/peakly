### Key Observations

1. **app.jsx at 1.30 MB is the bottleneck.** Babel Standalone must parse this on every cold load. On mid-range mobile (3G), this could mean 4-6 second initial parse. The service worker's stale-while-revalidate strategy mitigates repeat visits, but first-load UX on slow connections is rough.

2. **Service worker is healthy.** `CACHE_NAME = "peakly-v10"`, precaches `app.jsx`, network-first for `index.html` (prevents stale HTML issues from the previous outage), stale-while-revalidate for other assets. Architecture is sound.

3. **Weather cache confirmed working.** localStorage with 30-min TTL. At 2,226 venues batched 50/request with 2s delays, a cold load fires ~90 API calls. Cache absorbs repeat loads. Free tier (10K/day) supports ~100+ cold loads/day.

4. **Flight price fetching is well-engineered.** Deduplication by airport code (~100-150 unique), semaphore caps at 3 concurrent, 5s timeout with AbortController, exponential backoff on 429/5xx. No changes needed.

5. **No image lazy loading.** 2,226 venues with Unsplash `w=800&h=600` photos. Browser won't fetch all at once (only visible cards render), but adding `loading="lazy"` to `<img>` tags would be a cheap win.

### Performance Verdict

No regression from yesterday. The 1.3 MB JSX file remains the elephant in the room, but the single-file architecture is a deliberate design choice. Service worker caching makes this acceptable for repeat visitors. First-visit mobile performance is the weak link.
