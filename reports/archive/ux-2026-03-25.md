## 3. VISUAL QUALITY AUDIT

### New Issues Found This Cycle

**Issue N1: Emoji in UI chrome violates design decision**
CLAUDE.md states: "Remove emoji from UI chrome. Emoji in category pills, section headers, and badges look amateur. Clean text + SVG only." But VenueDetailSheet still uses emoji in section headers:
- Line 5080: `💡 You'd also like`
- Line 5107: `🎯 Insider Tips`
- Line 5122: `🛍️ Gear for this trip`
- Line 5146: `🎟️ Book an experience`
- Line 5188: `🏨` in Booking.com CTA
- Line 5200: `🛡️` in SafetyWing CTA
- Line 5213: `⚡ Peakly Pro`
- Line 5237: `📂 Save to list`
- Lines 5221–5224: Feature list uses emoji bullets
- Line 4969: `📤 Share & Invite` button
- Line 4976: `📍` location prefix

That's 15+ emoji instances in the primary conversion surface. This directly contradicts the "clean text + SVG only" decision. The BottomNav correctly uses SVG icons — VenueDetailSheet should follow suit.

**Issue N2: Similar venues hero thumbnails don't use photos**
Line 5084: Similar venue cards fall through to `sv.gradient` background instead of using `sv.photo`. The code checks `sv.photo` but the `height:62` thumbnail makes photos hard to see. More importantly, the photo exists but the gradient fallback path (line 5088) still renders if `sv.photo` is falsy — which shouldn't happen since all 205 venues have photos. Verify that `sv.photo` is being passed through the `listings` prop correctly.

**Issue N3: No loading state on VenueDetailSheet hero image**
Line 4953: The hero `<img>` has `loading="lazy"` but no shimmer/skeleton placeholder. On slow connections, users see a blank gray rectangle for 1–3 seconds before the photo loads. Airbnb shows a blurred thumbnail (LQIP). At minimum, show the gradient fallback until the image loads.

**Issue N4: Share panel emoji (line 4988)**
The share panel uses `✅`, `🔗`, `📋` emoji in buttons. These should be SVG icons for consistency with the design decision.

## 4. AIRBNB / COMPETITOR COMPARISON

### What improved
The VenueDetailSheet photo hero + sticky CTA now matches the structural quality of Airbnb's listing detail. Full-bleed hero, overlay controls, sticky booking bar. This was the biggest gap and it's closed.

### Top 3 remaining visual gaps

**Gap 1: Typography polish — Airbnb's type scale is tighter**
Peakly uses 16 different font sizes (8px to 88px). Airbnb uses ~8 sizes with stricter hierarchy. The VenueDetailSheet alone uses: 9, 10, 11, 12, 13, 14, 16, 20, 22px. This creates visual noise. Recommendation: consolidate to 10, 12, 14, 18, 22px for the detail sheet.

**Gap 2: Loading states — Airbnb shows progressive content**
Airbnb's listing detail loads with a shimmer skeleton matching the final layout shape. Peakly's VenueDetailSheet has no loading state — it pops in fully formed or not at all. The hero image specifically needs a placeholder.

**Gap 3: Section spacing rhythm — Peakly's is irregular**
VenueDetailSheet section spacing: marginBottom varies between 14px, 16px, and sometimes 10px with no clear system. Airbnb uses consistent 24px section gaps. Pick one value (16px) and apply it uniformly.

### Inspiration: Steal from Surfline
Surfline's spot detail page shows a **condition breakdown bar** — a horizontal stack of colored segments showing which factors contribute to the score (swell 30%, wind 25%, tide 20%, etc.). Peakly has all this data in `scoreVenue()` but never shows the breakdown. Adding a simple segmented bar under the condition score in VenueDetailSheet would make the score feel transparent and trustworthy — especially valuable since we just added thumbs up/down voting.
