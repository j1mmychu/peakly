### File Size
- **app.jsx**: 10,993 lines / ~1.95 MB raw
- After Babel transpilation overhead: estimated ~2.2–2.5 MB parsed in browser
- This is getting heavy for a single-file SPA. Not critical yet but approaching the threshold where code splitting would help.

### Venue Count
- **173 venues** currently in the VENUES array (note: CLAUDE.md claims 3,726 — there may be a discrepancy or the count was trimmed in a recent commit)

### What's Working Well
- Weather batching: 50 venues/batch, only top 100 fetched on load, lazy-fetch for detail views
- Flight price deduplication: ~250 potential calls reduced to ~15 via airport code mapping
- Native `loading="lazy"` on all images
- Pagination: 30 venues/page with "Show more" — prevents DOM bloat
- Weather cache: 2hr TTL (reduces API calls ~95%)
- Flight cache: 15min TTL with 2hr cleanup sweep

### Performance Concerns (Priority Order)
1. **797+ inline style objects** — every render creates new objects, triggering cascading child re-renders. This is the single biggest performance issue. Extracting to `useMemo()` or constants would cut re-renders significantly.
2. **50+ inline onClick handlers** — same issue, new function references on every render. Only 3 `useCallback` instances exist.
3. **Comment mismatch on line 4451** — weather TTL comment says "30 minutes" but code is 2 hours. Not a bug, just misleading documentation.

## Security Review: CLEAN ✅

### Good
- **Travelpayouts token**: Server-side only via `TRAVELPAYOUTS_TOKEN` env var. Not in any client code.
- **Proxy architecture**: Caddy → localhost:3001. VPS port not directly exposed.
- **CORS**: Properly restricted to `https://j1mmychu.github.io`, `http://localhost:8000`, `http://localhost:3000`.
- **No secrets in commits**: No API keys, passwords, or tokens found in app.jsx, index.html, or sw.js.

### Acceptable Risk
- **Sentry DSN exposed** in index.html (line 77) and app.jsx (line 8). This is standard for browser-side Sentry — the DSN is intentionally public. Recommendation: set rate limits and data scrubbing rules in Sentry project settings to prevent abuse.

### Missing (Non-Critical)
- **No rate limiting on proxy**: The Node.js proxy at `server/proxy.js` doesn't appear to have request rate limiting. Someone could hammer it. Consider adding `express-rate-limit` or Caddy-level rate limits.
