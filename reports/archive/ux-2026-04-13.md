## Top 3 Visual Issues

### 1. Photo overlay still crushing dark photos
Same as Apr 12. Screenshot any dusk/night venue — title becomes unreadable. Airbnb uses 0.25–0.35; we use up to 0.60.

**Fix** (4 one-liners):
```jsx
// app.jsx:1906 (ListingCard)
background:"linear-gradient(to top, rgba(0,0,0,0.38) 0%, transparent 52%)"
// app.jsx:2056 (FeaturedCard)
background:"linear-gradient(to top, rgba(0,0,0,0.40) 0%, transparent 55%)"
// app.jsx:2143 (CompactCard)
background:"linear-gradient(to top, rgba(0,0,0,0.40) 0%, transparent 50%)"
// app.jsx:3189 (Hero)
background:"linear-gradient(to top, rgba(0,0,0,0.35) 0%, transparent 60%)"
```

### 2. Type scale is 17 sizes from 9→72px
Count: 9, 10, 11, 12, 13, 14, 15, 16, 18, 20, 22, 24, 26, 28, 30, 32, 72. Apple SF stack uses ~7. Nothing feels like it belongs to the same system.

**Decision:** collapse to 8 tokens.
```js
const T = { xs:11, sm:12, base:13, lg:15, xl:18, display:22, hero:28, huge:40 };
```
Eliminates 9/10 (illegible), 14 (redundant with 13/15), 16 (redundant with 15), 20/24/26 (use 22 everywhere). One-time find-replace pass, ~250 sites.

### 3. `.card:active` scales to 0.95 — cards physically shrink 5% on tap
Line 89. That's *huge*. Apple HIG and material press states are 0.97–0.99. Combined with `.pill:active { scale(0.88) }` and `.tab-btn:active { scale(0.88) }`, everything rubber-bounces on touch. Feels like a toy.

**Fix:**
```css
.card:active { transform: scale(0.98); }
.pill:active { transform: scale(0.96); }
.tab-btn:active { transform: scale(0.94); opacity: 0.7; }
.pressable:active { transform: scale(0.97); }
.heart:active { transform: scale(1.15); }  /* was 1.35 */
```

## Top 3 Interaction Issues

### 1. Tap targets below 44×44 all over
- Share button `ListingCard` L1910: 32×32 → bump to 40×40, or drop share entirely from the card (it's noise — the card itself is a tap target, share belongs in the detail sheet)
- Heart button L1924: 36×36 → 44×44
- Heart button `CompactCard` L2144: 36×36 at `top:2, right:2` with *no padding* → 40×40 and top:6 right:6
- Category pill L3047: padding `5px 6px`, fontSize 11 → ~32px tall. Bump to `9px 10px`, fontSize 12, `minHeight:40`
- Ski-pass pill L3080: padding `5px 12px` → same issue
- Bottom nav button: icon 20px + label 10px + 8px vert padding = ~48px, OK, but label font 10 is illegible at arm's length → bump to 11

### 2. Hero card is the app's first impression — still shows shimmer strip + "WAIT" badge while weather loads
Still unfixed from Apr 12. New users land, see the hero card with `<div className="shimmer">` in the conditions box and no verdict badge for ~3s. That's the #1 recommendation telling them "data not ready."

**Fix — show nothing until hero has real data, then animate in:**
```jsx
// In ExploreTab, gate hero render on weatherLoaded
{!loading && heroPick && heroPick.conditionLabel !== "Checking conditions…" && !showSaved && (() => {
  // existing hero code
})()}
// Above the hero, while waiting:
{!loading && heroPick && heroPick.conditionLabel === "Checking conditions…" && (
  <div style={{margin:"12px 14px 0"}}><SkeletonCard /></div>
)}
```

### 3. Bottom nav active state is a 4×4px dot — invisible
Line 6445. Below the label, 4×4px, same color as the label text. On a 393px iPhone screen this dot is smaller than the tap-highlight radius. Users can't tell which tab they're on.

**Fix — turn the label blue + bold, kill the dot:**
```jsx
<span style={{ fontSize:11, fontWeight: active===t.id ? 800 : 500, fontFamily:F }}>{t.label}</span>
// and remove the <div style={{ width:4, height:4, ...}} /> below it
```

## Inspiration

**Surfline's session rating uses a single 4-bar block** (Poor/Fair/Good/Epic), color-filled left-to-right. Glance = know. No "72/100" math.

Peakly's `GoVerdictBadge` already has the label ("GO"/"WAIT"/"SKIP") — the numeric score next to it on the hero is redundant and makes users do arithmetic. Either:
- Keep the number, drop the label. OR
- Keep the label, drop the number. OR
- Show 5 dots filled proportionally (my pick — no number ambiguity, works in any language, matches Surfline's glanceability).

This also solves the CLAUDE.md onboarding open-issue #1: "no scoring explanation." A 5-dot meter needs no explanation.

## Decision Made

**Kill the "LIVE" badge on flight prices.** (`ListingCard`, lines 2072-2082 in various cards.) "LIVE" next to `$380` reads like breaking-news TV chyron, not a flight price. Users don't need to know data freshness — they need to know whether it's a deal. The `{pct}% below typical` pill *already* signals real data. "LIVE" is noise.

Replace with: nothing. If flight is live, show price in `#222` bold. If estimate, show `~$380 typical` in `#717171`. That's it. The visual weight tells the story.
