# Peakly UX Audit — 2026-03-28

## Design Score: 7.0/10

Score drops 0.2 from yesterday's 7.2. The codebase is 9,036 lines now — nearly double the documented ~5,413 — and UX debt is compounding. The same issues flagged across 4 consecutive reports (WCAG contrast, emoji in chrome, category pill touch targets, scroll-to-top on category switch) remain unfixed. Today's audit digs into three new findings alongside the persistent backlog.

---

## Top 3 Visual Issues

### 1. Category pill touch targets are dangerously small (REPEAT — 4th report)

**FILE:** app.jsx
**LINE:** 5253
**SEVERITY:** High (accessibility + usability)

Category pills use `padding: 5px 6px` with `fontSize: 11`. On a 375px-wide iPhone, the collapsed 4-pill layout gives each pill roughly 80px × 28px. Apple HIG minimum is 44px height. Android Material spec is 48px. We're at 28px — 36% below the floor.

Users with larger fingers will mis-tap constantly, especially switching between "Ski/Board" and "Surf" which sit adjacent.

**FIX (surgical):**
```jsx
// Line 5253–5260: increase pill padding
padding: "8px 10px",  // was "5px 6px"
fontSize: 12,          // was 11
```

This gets pills to ~38px height — still below 44px but significantly improved. True fix requires reducing the number of visible pills to 3 (not 4) in collapsed mode.

### 2. Book button color inconsistency across card types

**FILE:** app.jsx
**LINES:** 4249 (ListingCard), 4321 (FeaturedCard), 4455 (hero CTA)

ListingCard and FeaturedCard "Book" buttons use `background: linear-gradient(135deg, #1a56db, #0ea5e9)` — a blue gradient. But the hero card's primary CTA ("View Details") uses `background: #0284c7` — a flat sky blue. The design decision documented in CLAUDE.md says CTA buttons should be `#222` (dark). Three different button styles across three card types breaks visual consistency.

**FIX:** Standardize all primary CTAs to `#222` per the documented design decision:
```jsx
// All card Book/CTA buttons:
background: "#222",
```

### 3. "est." label missing from ListingCard flight prices (REPEAT — 3rd report)

**FILE:** app.jsx
**LINE:** 4183–4184 (price pill overlay on ListingCard)

ListingCard shows `${listing.flight.price}` with no indication when the price is an estimate vs. live. CompactCard correctly shows "est." at line ~4408. ListingCard is the primary card type users see in the Explore grid — this is the most important place to show the distinction.

**FIX:**
```jsx
// Line 4184, after the price span:
{!listing.flight.live && (
  <span style={{ fontSize: 8, color: "#888", fontWeight: 600 }}> est.</span>
)}
```

## Top 3 Interaction Issues

### 1. No scroll-to-top on category switch (REPEAT — 3rd report)

**FILE:** app.jsx
**LINE:** 5250
**COMPONENT:** ExploreTab pill onClick

When switching categories, `visibleCount` resets to 30 but `scrollRef.current` keeps its position. Users who scrolled 80+ venues deep in one category land mid-list in the new one. Disorienting.

**FIX:**
```jsx
// Line 5250, inside the pill onClick:
onClick={() => {
  setActiveCat(c.id);
  setVisibleCount(30);
  if (c.id !== "skiing") setSearch(s => ({...s, skiPass:""}));
  scrollRef.current?.scrollTo({ top: 0, behavior: 'instant' });  // ADD THIS
  haptic();
}}
```

### 2. BottomNav touch targets undersized

**FILE:** app.jsx
**LINE:** 8551
**COMPONENT:** BottomNav

Tab buttons use `padding: 8px 0` with no explicit width. The tappable area depends entirely on the icon (20px) + label (10px font) + gap. Effective tap area is roughly 40×40px — just below the 44px minimum. For bottom nav, which users tap hundreds of times per session, this needs to be generous.

**FIX:**
```jsx
// Line 8555: increase padding and add minWidth
padding: "10px 20px",
minWidth: 64,
minHeight: 44,
```

### 3. No loading skeleton for "Show more" pagination

**FILE:** app.jsx
**LINE:** ~5591
**COMPONENT:** ExploreTab "Show more" button

Tapping "Show more" renders 30 additional cards synchronously. On mid-range phones with 2,226 venues, this causes visible jank (200-400ms paint delay). React 18's `startTransition` is available and would fix this.

**FIX:**
```jsx
// Wrap the setVisibleCount in startTransition:
onClick={() => {
  React.startTransition(() => setVisibleCount(v => v + 30));
}}
```

## Typography & Color Audit

**Type scale — mostly consistent:**
- Headers: 18-24px, weight 800-900 ✓
- Body: 12-14px, weight 600-700 ✓
- Micro: 9-11px, weight 600-800 ✓
- Hierarchy is clear. Plus Jakarta Sans at these weights works well.

**Color palette — too many competing blues:**
- Primary: `#0284c7` (sky-600)
- Book button gradient: `#1a56db` → `#0ea5e9` (different blue)
- Hero CTA: `#0284c7` (flat)
- Price text: `#0284c7`
- Alert banner gradient: `#0284c7` → `#38bdf8`

Four distinct blue treatments. Should be one. Recommendation: standardize interactive elements to `#222` (dark) per CLAUDE.md design decision, reserve `#0284c7` for informational accents (links, score colors).

**Contrast issues (WCAG AA failures):**
- Line 4875: `fontSize:11, color:"#717171"` on white = 4.48:1 (AA requires 4.5:1 for small text — FAIL at 11px)
- Line 5288: `fontSize:10, color:"#999"` on white = 2.85:1 (FAIL)
- Line 4907: `color:"rgba(255,255,255,0.82)"` on gradient = variable, likely fails
- Line 6237: `color:"rgba(255,255,255,0.5)"` on dark = 3.02:1 (FAIL)

## User Flow Analysis

### Primary flow: Land → Browse → Tap venue → View details → Book flight

1. **Land** (splash → Explore tab): Smooth. Splash is 1.8s minimum, then fades. First venues render after first batch of 50 weather fetches. Good.
2. **Browse** (scroll grid): Cards render well with photos. Shimmer loading states work. Score badges are clear.
3. **Tap venue**: Opens VenueDetailSheet with sheet-up animation (0.44s spring curve). Smooth.
4. **View details**: Photo hero + sticky CTA at bottom. Weather data, similar venues, tips. Well-structured.
5. **Book flight**: Opens Aviasales in new tab. **TP_MARKER is still "YOUR_TP_MARKER"** — zero commission on every click. This is a revenue bug, not a UX bug, but it's been flagged since 2026-03-27.

**Tap count to book: 3** (tap venue card → scroll to CTA → tap Book). This is good. Airbnb is also 3. Could be 2 if the ListingCard "Book" button went directly to Aviasales (it does — so technically 1 tap if user clicks Book directly on the card).

### Secondary flow: Change home airport → See prices update

Works. Profile → Edit → type airport → autocomplete → select. Prices re-fetch via `useEffect` dependency on `profile.homeAirport`. Takes 2-5s for new prices to populate depending on API response times. No loading indicator during price refresh — user sees stale prices until new ones arrive. Should show a subtle shimmer on price elements during refresh.

## Inspiration

**Steal from Airbnb: the "Total before taxes" price toggle.** Peakly shows flight prices but never clarifies whether they're one-way or round-trip. Users have no idea. Airbnb added a simple toggle between "price per night" and "total before taxes" that dramatically improved trust. Peakly should add a small "(round trip)" or "(one way)" label under every price display. One line of code per card component. Massive trust improvement.

---

## Decision Made

**All CTA buttons standardize to `#222` (dark) immediately.** The gradient blues on Book buttons (`linear-gradient(135deg,#1a56db,#0ea5e9)`) compete with the informational blue (`#0284c7`) used for scores, links, and accents. CLAUDE.md already documents this decision but it was never fully applied to card components. Dark CTAs feel more premium and create clear visual hierarchy: dark = action, blue = information.

---

## Summary

The app looks good at a distance but falls apart under scrutiny. Seven issues have been flagged and re-flagged across 4 daily reports with zero fixes applied. The scroll-to-top fix is literally one line of code. The pill padding fix is changing two numbers. These aren't architecture problems — they're attention problems.

**If I could ship only one fix today:** scroll-to-top on category switch. It's the single most disorienting interaction in the app and it's a one-line change.

**Design Score: 7.0/10** — down from 7.2 because compounding unfixed issues erode the experience over time, not because anything got worse. The foundation is solid. The polish layer needs actual deployment.
