## Performance Analysis

| Metric | Value | Assessment |
|--------|-------|------------|
| app.jsx lines | 10,976 | Large single file — manageable for Babel Standalone |
| app.jsx size | ~1.95 MB (1,999,810 bytes) | **CRITICAL** — approaching 2MB raw JSX |
| index.html | 14.6 KB | Normal |
| sw.js | 3.1 KB | Normal |
| manifest.json | 1.2 KB | Normal |
| Total frontend payload | ~2.02 MB (before Babel) | Heavy for mobile |

### Performance Concerns

1. **2MB JSX file is the #1 bottleneck.** Babel Standalone must parse and transpile ~2MB of JSX on every page load. On a mid-tier mobile device, this can take 2-5 seconds before any React rendering begins. This is the single biggest performance risk.

2. **3,726 hardcoded venues in the JS bundle.** The VENUES array alone is likely ~1.2-1.4MB of the file. This data ships to every user on every load, even if they only browse 30 venues (the pagination limit).

3. **Batched weather fetching (50/batch, 2s delay)** is well-designed for API limits but means full weather data takes ~150 seconds (2.5 min) to load for all 3,726 venues. The progressive loading strategy mitigates this for UX.

4. **No code splitting possible** in the current architecture (single file, no bundler). This is a known tradeoff per CLAUDE.md — accepted for simplicity.

### Estimated Page Load Timeline (3G mobile)
- Download 2MB JSX: ~3-4s
- Babel transpile: ~2-4s
- React initial render: ~0.5-1s
- First meaningful paint: **~6-9 seconds**
- Weather data fully loaded: ~3+ minutes

**Decision: No action now.** The single-file architecture is a deliberate choice. If load times become a user complaint post-Reddit launch, the lowest-effort optimization would be extracting VENUES data into a separate JSON file loaded via fetch — keeps the no-build-step philosophy while cutting ~1.2MB off the initial JSX parse.

## Decision Made

**Adding `express-rate-limit` to the proxy is the next infrastructure priority.** Without it, a single bad actor can burn through the entire Travelpayouts API quota in minutes. The fix is trivial:

```js
const rateLimit = require('express-rate-limit');
app.use('/api/', rateLimit({ windowMs: 60000, max: 60 }));
```

This protects the API quota without affecting normal users (who make ~5-10 requests per session).
