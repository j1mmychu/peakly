## Top 3 Visual Issues

### 1. The dark CTA rule exists on exactly ONE button

Design decision from 2026-03-23: "CTA buttons: `#222` (dark, NOT blue)." In practice, this rule is honored on exactly one button in the entire app: the sticky Flights CTA inside `VenueDetailSheet` (L5909).

Every other primary CTA is blue or gradient:

| Location | Color | Line |
|----------|-------|------|
| ListingCard "Book" | `linear-gradient(135deg,#1a56db,#0ea5e9)` | L2124 |
| FeaturedCard "Book" | `linear-gradient(135deg,#1a56db,#0ea5e9)` | L2209 |
| Hero card "View Details" | `#0284c7` | L3342 |
| Alerts empty "Create alert" | `#0284c7` | L4037 |
| Onboarding final "Show me what's firing" | `linear-gradient(135deg,#0284c7,#38bdf8)` | L5404 |
| Profile "Redo Onboarding" | `linear-gradient(135deg,#0284c7,#38bdf8)` | L4238 |

The onboarding final CTA is the most damaging. It's the moment a new user commits to the app — and the button that fires is blue-gradient while the detail sheet CTA they'll see seconds later is dark. The visual language of "confirm / go / book" is incoherent from session start.

**Fix — 6 lines, 10 minutes:**
```jsx
// L2124 (ListingCard):
background:"#222", borderRadius:20, padding:"8px 14px", minHeight:44,

// L2209 (FeaturedCard):
background:"#222", borderRadius:20, padding:"8px 14px", minHeight:44,

// L3342 (Hero card "View Details"):
background:"#222", border:"none", borderRadius:10, padding:"10px 0",

// L4037 (Alerts empty state):
background:"#222", border:"none", borderRadius:14,

// L5404 (Onboarding final CTA — keep gradient as reward, acceptable exception):
// This one can stay blue — it's a one-time moment, not a recurring UI element.
// But remove the boxShadow glow at L5407: delete `boxShadow:"0 4px 20px rgba(2,132,199,0.4)"`
```

### 2. "Best Right Now" carousel images have no fade-in

All other card photos in the app use the `opacity:0 → onLoad → opacity:1` pattern with a `0.35s` transition. The `bestRightNow` carousel (L3395–3425) renders `<img>` elements without any `opacity` start state or `onLoad` handler. Photos in this carousel pop in hard — no transition. On a slow connection, the color gradient background flashes away and the photo slams in.

The fix is one line per image. The pattern is already established elsewhere in the codebase.

```jsx
// L3404 — bestRightNow carousel img:
// Add:
ref={img => { if (img && img.complete && img.naturalWidth > 0) img.style.opacity = 1; }}
onLoad={e => { e.target.style.opacity = 1; }}
// Add to style prop:
opacity:0, transition:"opacity 0.35s ease"
```

### 3. Flight price color is blue in carousel, dark in cards, green in detail

The price number uses three different colors across three card surfaces:

- `bestRightNow` carousel (L3415): `color:"#0284c7"` (sky blue)
- `ListingCard` (L2111): `color:"#222"` (dark)
- `VenueDetailSheet` flight panel (L5685): `color:"#16a34a"` (green)

A user who taps from the carousel to the detail sheet sees the price change color mid-flow. This is not a color system — it's three people who didn't talk to each other.

**Decision rule:** Flight price = `#222` everywhere. The context (card, carousel, detail) already disambiguates what the number means. Color does not need to do additional signaling.

```jsx
// L3415 — carousel price: change #0284c7 → #222
// L5685 — detail panel price: change #16a34a → #222
```

## Top 3 Interaction Issues

### I1. Share button: 32px, wrong icon — 6-day carry-over

L2011: `width:32, height:32`. Apple HIG minimum: 44px. The icon is `↑` (up arrow U+2191) — not a share glyph on any platform. Users either miss-tap it or don't know what it does. Minimum fix: `width:38, height:38`. Real fix: replace `↑` with an SVG share icon.

### I2. Airport change doesn't surface price update confirmation

Flow: Profile tab → airport field → select airport. Prices update reactively — that part works. But there's no visual acknowledgment. The user changes their airport, prices recalculate in the background, and nothing confirms the change landed. On a 3G connection where flight data takes 2–4 seconds, users can't tell if the new prices are from the new or old airport.

A subtle toast ("Prices updated from LAX") would close this gap. One `useState` + a `useEffect` watching `profile.homeAirport` changes.

### I3. Tap count to book is 2 — this is good, protect it

Land → tap card (1) → tap Flights sticky CTA (2) = booking initiated. This is excellent. Airbnb takes 4+ taps to initiate booking. Don't let new features push this above 3. The current architecture with the always-visible sticky CTA is the right call.

**Warning:** The "Set Alert" button (L5690) sits above the 7-day forecast which is above the hotel/booking CTA. On small viewports, users scroll past the alert CTA before reaching the booking action. Consider moving "Set Alert" to below the forecast, or into the sticky bar as a secondary action alongside Flights and Hotels.

The execution just hasn't caught up.
