## 3. Weather & External APIs

**Status: ⚠️ YELLOW — rate limit is a scaling trap**

| Metric | Value | Concern Level |
|--------|-------|---------------|
| API endpoint | `api.open-meteo.com/v1/forecast` | Free tier |
| Free tier limit | 10,000 calls/day | Hard cap |
| Cache TTL | 30 minutes ✅ | Working |
| Cache cleanup | 2-hour max age ✅ | Working |
| Venues | 2,226 | High |
| Unique lat/lon (est.) | ~800–1,000 | Per cold user |
| Calls per cold load | ~1,000–2,000 (weather + marine) | Dangerous |

**The math on rate limits:**

```
2,226 venues ÷ 50 per batch = ~45 batches
Each batch: up to 50 simultaneous weather calls + marine calls for coastal venues
Conservative estimate: 1,500 unique API calls per fully-cold user load

10,000 free calls/day ÷ 1,500 = ~6 fully-cold user loads before hitting the cap
```

**localStorage cache is per-user.** It does NOT share between users. Cache works great for a returning user refreshing, but does nothing for a new user or someone clearing storage.

**At 10 daily active new users: ~15,000 API calls/day → OVER LIMIT.**

Open-Meteo's free tier will start returning 429s. The weather fetch has no explicit 429 handler — it throws `Error("weather fetch failed")` and venues will show "Checking conditions…" indefinitely.

**Fix — Add 429 handling to fetchWeather (app.jsx ~line 2990):**

```javascript
async function fetchWeather(lat, lon) {
  const cacheKey = _wxCacheKey("weather", lat, lon);
  const cached = _wxCacheGet(cacheKey);
  if (cached) return cached;
  const r = await fetch(url);
  if (r.status === 429) {
    console.warn("[Peakly] Open-Meteo rate limited — returning null");
    return null;  // caller handles null gracefully; shows "Forecast unavailable"
  }
  if (!r.ok) throw new Error("weather fetch failed");
  // ...
}
```

**Fix — Extend cache TTL from 30 min to 60 min** (reduces cold-load API calls by 2× for returning users):

```javascript
// app.jsx line 2950 — change:
const WX_CACHE_TTL = 60 * 60 * 1000; // 60 minutes (was 30)
```

**Longer fix — only batch-fetch visible venues, lazy-load the rest.** This is a 2-hour dev task but cuts cold-load API calls by 95%. Fetch weather for 30 visible venues first, then fetch the rest on demand when users scroll to them.

## What Breaks First at Scale

**Open-Meteo is the first thing that breaks.** It will fail at ~6+ daily new users. Here's the sequence: (1) rate limit hit → 429s → weather shows "Checking conditions…" → users see no scores → Explore tab looks broken → bounce rate spikes → no one converts. The fix is 10 minutes of code plus either throttling venue fetching to visible-only (2 hrs dev) or upgrading to Open-Meteo's commercial API (€29/mo, unblocks at 100K calls/day). Do the 429 handler now. Plan the commercial tier when DAU hits 20.

**Second failure: DigitalOcean VPS is a single point of failure.** No HA, no failover. A droplet reboot (monthly maintenance, kernel updates) takes the flight proxy offline for 30–90 seconds. At current user count this is acceptable. At 10K MAU this needs a health check + auto-restart via systemd and an UptimeRobot alert with <1 minute check interval (currently 5 min).

```bash
# On VPS — ensure flight proxy auto-restarts on crash:
sudo systemctl enable peakly-proxy
sudo systemctl start peakly-proxy
# Verify:
sudo systemctl status peakly-proxy
```
