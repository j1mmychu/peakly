**P1: Open-Meteo rate limit math is broken at scale.**

You have 3,726 venues. Each venue potentially triggers 2 API calls (weather + marine). Open-Meteo free tier: 10,000 calls/day = ~416 calls/hour = ~7 calls/minute.

Scenario: 100 concurrent users open the app simultaneously (entirely plausible after a Reddit post). Each user cold-loads the hero card + first 30 venues. That's potentially 60 calls per user × 100 users = **6,000 calls in under 60 seconds.** Open-Meteo will start returning 429s. Your app degrades gracefully (no scores shown), but this is silent failure — users see "—" scores and bounce.

Current cache is per-user browser localStorage. With 100 users, you have 100 separate cold caches. No cross-user sharing.

**The fix that costs $0 and takes 2 hours:**

Add a server-side weather cache to your VPS proxy. Cache Open-Meteo responses for 30 minutes. All users share one cache. 100 concurrent users → 1 API call per venue, not 100.

```javascript
// In server/proxy.js — add weather proxy endpoint

const wxCache = new Map(); // { key: { data, expires } }
const WX_TTL = 30 * 60 * 1000;

app.get('/api/weather', async (req, res) => {
  const { lat, lon, type = 'forecast' } = req.query;
  if (!lat || !lon) return res.status(400).json({ error: 'lat and lon required' });

  const key = `${type}_${parseFloat(lat).toFixed(2)}_${parseFloat(lon).toFixed(2)}`;
  const cached = wxCache.get(key);
  if (cached && Date.now() < cached.expires) {
    return res.json(cached.data);
  }

  const base = type === 'marine'
    ? 'https://marine-api.open-meteo.com/v1/marine'
    : 'https://api.open-meteo.com/v1/forecast';

  // Forward all query params except 'type'
  const params = new URLSearchParams(req.query);
  params.delete('type');
  const url = `${base}?${params}`;

  try {
    const upstream = await fetchJson(url);
    const data = upstream.json;
    wxCache.set(key, { data, expires: Date.now() + WX_TTL });
    res.json(data);
  } catch (err) {
    res.status(502).json({ error: err.message });
  }
});
```

Then in `app.jsx`, change `METEO` and `MARINE` constants (lines 4449–4450):
```javascript
const METEO  = "https://peakly-api.duckdns.org/api/weather?type=forecast";
const MARINE = "https://peakly-api.duckdns.org/api/weather?type=marine";
```

This routes all weather calls through your VPS, shared cache, no rate limit risk.

**1.46 MB gzipped over a 3G connection (~1 Mbps) = 12 seconds to first meaningful paint.** On WiFi this is fine. On mobile data this is brutal.

**The #1 bottleneck: Babel Standalone (~850 KB gzipped, runs synchronously at parse time).** You are shipping a full JavaScript compiler to every user's browser to transpile JSX once. This is architecturally correct for a no-build setup but has a hard performance ceiling.

### Image Lazy Loading
- Total `<img>` tags in app.jsx: **8**
- All 8 use `loading="lazy"` ✅

However: venue photos in `ListingCard`, `CompactCard`, and `FeaturedCard` appear to be rendered as CSS `background-image` in `<div>` elements, not `<img>` tags. CSS background images **are not lazy loaded** by the browser. On the Explore tab with 30 visible cards, this triggers 30 simultaneous Unsplash image fetches on mount. Not catastrophic (Unsplash has a CDN), but adds ~200–400ms to perceived load time on mobile.

Fix — for venue cards with photo divs, use `<img loading="lazy">` with `object-fit: cover` instead of background-image where possible. Or add an Intersection Observer for the card components. This is a P3 — not blocking launch.

### Babel Version
`@7.24.7` — latest is `7.26.x`. Not critical, but `7.25.x` has parse performance improvements worth ~5–8% on large files. Update: change one line in `index.html`:
