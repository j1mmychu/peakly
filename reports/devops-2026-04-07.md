### Performance Concerns — P1

1. **app.jsx is now ~2MB.** The 3,726 hardcoded venue objects account for most of this. Babel Standalone must parse and transpile the entire file on every page load. On mid-range phones this is a 3-5 second blocking operation. This is the single biggest performance bottleneck.
2. **Batched weather fetching (50/batch, 2s delay)** — with 3,726 venues this means ~75 batches × 2s = ~150 seconds to fetch all weather. The pagination (30 venues visible) mitigates this since users only see a fraction, but background fetching still hammers the API.
3. **No code splitting possible** — single-file architecture means everything loads at once. The venue data alone (~1.5MB of JSON literals) could be extracted to a static JSON file loaded via fetch, cutting initial parse time significantly. **This is the #1 performance optimization available.**
