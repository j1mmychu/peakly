**Bottlenecks flagged:**

1. **Babel Standalone is ~800 KB** and blocks rendering. It transpiles 469 KB of JSX at runtime. On a slow mobile connection this is a 3–5 second white screen. Pre-transpiling to plain JS would cut initial load by ~60%, but this conflicts with the no-build-step architecture decision. Noted, not actionable yet.
2. **No image lazy loading** — venue photos are Unsplash URLs loaded eagerly. When scroll lists get long, this hammers bandwidth. `loading="lazy"` on `<img>` tags would be a surgical one-liner fix.
3. **39 fetch calls** — weather is batched 50 per 2s which is good. Flight data fetches per-venue on scroll could benefit from a request deduplication layer if users scroll fast.

## Security Review: CLEAN (minor items noted)

**Good:**
- Travelpayouts token is server-side only (`process.env.TRAVELPAYOUTS_TOKEN` in proxy.js, line 11). Not in `app.jsx` or any client-side file.
- `.gitignore` covers `.env`, `.env.*`, `*.pem`, `*.key` — solid.
- Proxy has rate limiting: 60 req/min per IP with cleanup every 5 min.
- IATA regex validation on flight endpoints prevents injection.
- CORS locked to specific origins (GitHub Pages, peakly.app, localhost).
- Alert endpoint has schema validation + 10K cap.
- Push token stored in localStorage as `peakly_push_token` — this is fine for web push registration tokens (not a secret).
- No `.env` files, no `.pem` files, no secret files found in the repo.

**Issues (medium priority):**

1. **No SRI (Subresource Integrity) on CDN scripts** — React, ReactDOM, Babel, and Sentry are loaded from unpkg/sentry-cdn without `integrity` attributes. A CDN compromise could inject malicious code. However, adding SRI to Babel Standalone may break `eval`-based transpilation. Recommend: add SRI to React + ReactDOM + Sentry (safe), defer Babel SRI research.
2. **No CSP (Content Security Policy) meta tag** — still missing per CLAUDE.md open item #3. Adding a strict CSP would break Babel's `eval()`, so this needs careful crafting (`'unsafe-eval'` required, which partially defeats the purpose). Low ROI until build step is adopted.
3. **Sentry DSN exposed in index.html** (line 77: `9416b032a46681d74645b056fcb08eb7`) — this is normal for Sentry (client-side by design, rate-limited on their end), but worth noting.
4. **proxy.js has duplicate `require` statements** — `fs` and `path` are required at both line 5-6 and line 232-233. Not a security issue but sloppy — Node.js caches modules so no runtime impact, but should be cleaned up.

## Decision Made

**Adding SRI hashes to React and ReactDOM scripts this week.** These are stable pinned versions (18.3.1) so SRI is safe and trivial. Sentry can also get SRI. Babel SRI deferred — needs testing since it `eval()`s user code.

```bash
# Generate SRI hashes:
curl -s https://unpkg.com/react@18.3.1/umd/react.production.min.js | openssl dgst -sha384 -binary | openssl base64 -A
curl -s https://unpkg.com/react-dom@18.3.1/umd/react-dom.production.min.js | openssl dgst -sha384 -binary | openssl base64 -A
```
