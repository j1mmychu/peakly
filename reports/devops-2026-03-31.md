| Pagination | 30 venues/page | 🟢 DOM performance managed |

**Reddit traffic concern:** If r/surfing or r/skiing posts hit front page, expect 2-5K concurrent visitors. The 2MB app.jsx Babel transpile is the bottleneck — every visitor re-parses 10K lines of JSX client-side. Service worker caches the transpiled result after first load, but first-time visitors will feel it.

**Recommendation:** Post-launch, consider a pre-transpiled build for production. For today, the SW cache mitigates repeat visits.

## Security Review: ⚠️ 2 ISSUES

### ✅ Clean
- No API tokens, secrets, or passwords in app.jsx or index.html
- Travelpayouts token is env-var only (`process.env.TRAVELPAYOUTS_TOKEN`)
- Proxy CORS locked to `j1mmychu.github.io` + localhost origins
- Proxy binds to `127.0.0.1` only (not exposed directly)

### 🔴 Issue 1: No rate limiting on proxy
The proxy server (`server/proxy.js`) has **zero rate limiting**. Any client can hammer `/api/flights/latest` and exhaust the Travelpayouts API quota. With Reddit traffic incoming, a single bad actor (or overeager client-side fetch loop) could burn through the API allowance.

**Fix (post-launch):** Add `express-rate-limit` — 60 req/min per IP on flight endpoints. One `npm install` + 5 lines of code.

### 🟡 Issue 2: In-memory alert store
`_alerts` is a JavaScript `Map()` — alerts vanish on PM2 restart or server crash. Not critical for launch (alert polling isn't live yet), but needs SQLite or flat-file persistence before alerts go live.
