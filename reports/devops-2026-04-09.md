The proxy itself is solid. HTTPS is live. Timeout and retry are correct. The one hole: there is zero rate limiting in `server/proxy.js`. Any script kiddie or competitor can call `/api/flights/latest?origin=LAX&destination=BKK` in a loop and exhaust the Travelpayouts daily quota in minutes. When quota is gone, every real user sees broken flight prices.

**Fix — add `express-rate-limit` to `server/proxy.js` (15 minutes + `npm install express-rate-limit` on VPS):**

```js
// Add to server/proxy.js after `const app = express();`
const rateLimit = require('express-rate-limit');

const flightLimiter = rateLimit({
  windowMs: 60 * 1000,      // 1 minute window
  max: 20,                   // 20 flight lookups per IP per minute
  standardHeaders: true,
  legacyHeaders: false,
  message: { success: false, error: 'Too many requests, slow down.' },
});

app.use('/api/flights', flightLimiter);
```

**VPS deploy commands:**
```bash

cd /path/to/peakly-proxy   # wherever proxy.js lives on the VPS
npm install express-rate-limit
pm2 restart peakly-proxy
```

| Check | Result | Status |

| Inter-batch delay | **None — both batches fire immediately** | YELLOW |

**No inter-batch delay in the code.** CLAUDE.md says "50/batch, 2s delay" but `fetchInitialWeather` (lines 10582–10615) has no `await` sleep between batches. Both batches fire in immediate sequence — effectively 100 concurrent API calls at app load. Open-Meteo free tier is 10,000 calls/day. At 68 new users/day (each firing 150 fresh calls), you hit the limit. The 30-min cache helps for returning users, but new-user spikes — like a Reddit post — will hammer this.

**At-scale math:**
- 100 fresh users in one day (conservative Reddit spike): 100 × 150 = **15,000 calls — 50% over free tier**
- Fix: add 1-second delay between batches. This has near-zero UX impact (second batch is background, UI already renders after batch 1).

**Fix (5 minutes):**
```js
// app.jsx — inside fetchInitialWeather, after the first batch results.forEach block
// Add this sleep before the loop continues:
for (let i = 0; i < initial.length; i += BATCH_SIZE) {
  const batch = initial.slice(i, i + BATCH_SIZE);
  const results = await Promise.allSettled(/* ... */);
  results.forEach(/* ... */);
  if (i === 0) {
    setWxData({...wxRef.current});
    setMarData({...marRef.current});
    setLoading(false);
    setWxLastUpdated(new Date());
  }
  // ← ADD THIS:
  if (i + BATCH_SIZE < initial.length) {
    await new Promise(res => setTimeout(res, 1000));
  }
}
```

**If Open-Meteo starts returning 429s:** the fix is `VENUES.slice(0, 50)` for the initial batch. Halves API load, UI still looks identical to users.

## 6. P0 — PUSH NOTIFICATION ICONS ARE BROKEN

This is the most embarrassing live bug. In `sw.js` lines 63–64:

```js
icon: "/peakly/manifest.json",
badge: "/peakly/manifest.json",
```

`manifest.json` is a **JSON file**, not an image. Every push notification Peakly sends displays a broken/missing icon. The OS tries to render a JSON file as a PNG and fails silently.

**Fix (2 minutes):**
```js
// sw.js — replace lines 63-64
icon: "data:image/svg+xml," + encodeURIComponent('<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 192 192"><rect width="192" height="192" rx="36" fill="%230284c7"/><text x="96" y="142" font-size="120" font-family="system-ui" font-weight="700" fill="white" text-anchor="middle">P</text></svg>'),
badge: "data:image/svg+xml," + encodeURIComponent('<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 96 96"><rect width="96" height="96" rx="18" fill="%230284c7"/><text x="48" y="71" font-size="60" font-family="system-ui" font-weight="700" fill="white" text-anchor="middle">P</text></svg>'),
```

Or use a proper hosted PNG. The inline SVG data URI is the zero-infrastructure fix.

After this fix, bump the SW cache name so users get the update:

## 7. P2 — ALERTS ARE IN-MEMORY ONLY (SERVER RESTART = LOST)

`server/proxy.js` line ~175:
```js
const _alerts = new Map(); // in-memory store (replace with DB later)
```

Every `pm2 restart`, server crash, or VPS reboot wipes all registered push alerts. Users who set alerts will silently stop receiving them after any maintenance.

**Fix options (ranked by effort):**
1. **Immediate (5 min):** Persist to a JSON file on disk. Good to ~10K alerts.
   ```js
   const ALERTS_FILE = '/var/data/peakly-alerts.json';
   let _alerts = new Map();
   try {
     const raw = require('fs').readFileSync(ALERTS_FILE, 'utf8');
     _alerts = new Map(Object.entries(JSON.parse(raw)));
   } catch {}
   
   function saveAlerts() {
     require('fs').writeFileSync(ALERTS_FILE, JSON.stringify(Object.fromEntries(_alerts)));
   }
   // Call saveAlerts() after every _alerts.set() and _alerts.delete()
   ```
2. **Proper (1 hour):** SQLite via `better-sqlite3`. Zero network dependency, single file.

## What Breaks First at Scale

**Open-Meteo is the single highest-probability outage vector.** At 70+ new users per day with cold localStorage caches (e.g. the day after a Reddit post), 10,500 API calls hit against a 10,000/day limit. Weather scores stop loading. The entire value proposition — "real-time condition scoring" — silently returns zeros. The app appears to work but all venues score identically. Users can't tell why. They churn.

The inter-batch delay fix (P1.2 above) reduces concurrent load. The `VENUES.slice(0, 50)` fallback (one line) cuts load in half if 429s appear. Both should be in place before the Reddit launch. After 1K users, upgrade to Open-Meteo commercial ($10–40/month) — it's the cheapest line item in the budget relative to what breaks without it.

Second priority: the proxy rate limiter (P0.1). Without it, any one bad actor can exhaust Travelpayouts quota silently — no error, no alert, just $0 flight affiliate revenue for every real user until quota resets at midnight.
